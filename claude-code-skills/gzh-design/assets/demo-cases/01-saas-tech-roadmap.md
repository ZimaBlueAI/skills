# 从单点脚本到平台化能力：一家 SaaS 团队的自动化路线图

> 自动化不是把人从流程里拿走，而是把人从重复确认里解放出来。

## 为什么现在必须重构

过去一年，我们的交付流程从 3 个客户扩展到 27 个客户。最初的脚本还能解决问题，但现在每次上线都需要人工检查配置、人工合并报表、人工同步状态。真正拖慢团队的不是代码，而是散落在各处的状态。

当前流程里有三个瓶颈：配置模板不可追踪，执行结果不可复现，失败原因不可聚合。如果继续叠加脚本，短期看似更快，长期会让每次发布都变成一次猜谜。

## 路线图一：统一任务模型

第一步不是上平台，而是把任务抽象清楚。每个自动化任务都应该有输入、执行器、输出、失败策略和负责人。只有任务模型统一，后续的调度、审计和权限才有共同语言。

| 模块 | 当前状态 | 目标状态 |
|---|---|---|
| 配置 | 人工复制 | 模板化参数 |
| 执行 | 本地脚本 | 托管运行 |
| 结果 | 群消息 | 可查询记录 |

## 路线图二：把失败做成资产

失败日志不应该只是报警。我们计划把失败原因聚类，形成可搜索的错误知识库。一个失败样本如果被修过三次，就应该进入规则库，而不是继续靠人记住。

关键指标是：重复失败率下降 40%，平均定位时间从 30 分钟降到 8 分钟以内。

## 路线图三：保留人工接管点

真正可靠的平台不是全自动，而是知道什么时候停下来。涉及客户数据、权限变更和跨系统写入时，流程必须保留人工确认。自动化负责准备上下文，人负责做判断。

## 结语

下一步，我们会先把发布检查这一个场景平台化。范围足够小，价值足够明确，也最容易验证团队是否真的准备好从脚本走向平台。
